# Avatar Recovery And Page Speed Design

## Goal

在不破坏数据库结构和历史头像数据的前提下，恢复线上头像可用性，并显著降低问答/辩论详情页的首屏请求数与渲染压力。

## Context

- 当前 `users.avatar` 已逐步收敛为 `/api/uploads/...` 路径，但 `backend` 容器没有持久化 `uploads` 目录，导致容器重建后数据库路径仍在、文件实际丢失，页面头像全部 404。
- `DebatePage`/`QuestionPage` 详情页会一次性拉较多回答，而 `AnswerItem` 在登录态下又会为每条回答单独请求收藏状态，形成明显的 N+1 请求。
- 本次修复的约束是“不破坏数据库”，因此不能依赖强制迁移或清库。

## Approaches Considered

### 方案 A：低风险分阶段修复

- 先修复头像文件持久化与裂图兜底
- 再做回答页请求和渲染降载
- 最后补充批量收藏状态接口，去掉 N+1

优点：风险最低，线上收益最快，兼容历史数据。  
缺点：性能收益分两步体现。

### 方案 B：一次性重做头像和回答页数据契约

- 引入全新的头像上传/元数据体系
- 同时重做回答详情页接口返回结构

优点：最终形态更统一。  
缺点：改动过大，不符合当前低风险止血目标。

### 方案 C：只做前端懒加载

- 不动后端，只减少首屏回答数和部分请求

优点：改动小。  
缺点：头像裂图无解，登录态 N+1 也无法彻底消除。

## Recommended Design

采用方案 A。

### 1. 头像恢复

- 为 `backend` 增加 `uploads` 目录持久化挂载，保证容器重建后头像文件仍然存在。
- 前端头像组件增加错误回退能力：当路径文件缺失时自动回退到默认头像，不再出现大面积裂图。
- 保持 `users.avatar` 兼容三类值：
  - 外链 URL
  - `/api/uploads/...` 路径
  - 历史遗留但尚未归一化的 base64/data URL

### 2. 页面级提速

- 将 `DebatePage` 首屏回答拉取数量从 50 降到更合理的 20。
- 保持问题详情与辩论详情的“显式继续加载”模式，不做数据库结构变化。
- 头像失败回退后，也减少浏览器连续 404 导致的观感卡顿。

### 3. 接口级提速

- 新增回答收藏状态批量查询接口，接收多个 `answer_id`，一次返回当前用户对这些回答的收藏状态。
- `DebatePage` 与 `QuestionPage` 在拿到回答列表后，只触发一次批量收藏状态查询。
- `AnswerItem` 支持外部传入初始收藏状态，默认不再自行逐条请求，避免 N+1。

## Error Handling

- 头像文件不存在时，页面应继续正常渲染，并自动使用默认头像。
- 批量收藏状态接口失败时，页面只降级为“未加载收藏状态”，不阻塞回答渲染。
- 对未登录用户不发收藏状态批量请求。

## Testing Strategy

- 后端为批量收藏状态接口补单测/回归测试。
- 前端构建校验类型与模板。
- 手工验证：
  - 头像路径存在时正常显示
  - 头像路径缺失时回退默认头像
  - `debate/:id` 与 `question/:id` 登录/未登录加载速度改善
  - 网络面板确认不再出现“每条回答一个收藏状态请求”
